## Issue: API Abuse in `get_stock_info` — Full List Fetched Per Stock

- **Layer**: 1
- **Severity**: Critical
- **Root Cause**: `FinMindClient.get_stock_info()` (line 77–86 of `src/data/finmind_client.py`) calls `self._loader.taiwan_stock_info()` which fetches the **entire** Taiwanese stock universe (~2000+ rows) on every invocation, then filters by `stock_id`. The cache key includes `stock_id`, so each unique stock ID triggers a separate full API call. When `get_stock_name()` (line 233) and `get_industry()` (line 240) are both called (as happens in `_router_base.py` line 13–18), the full list is fetched **three times** for a single stock page load.
- **Proposed Fix**:
  1. **Separate the "full list" cache from the "single stock" cache.** Introduce a dedicated method `_fetch_all_stock_info()` that fetches and caches the entire `taiwan_stock_info` table under a single cache key (e.g., prefix `"all_stock_info"` with no `stock_id` param).
  2. **Refactor `get_stock_info()`** to call the shared full-list cache first, then filter locally — never call the API per-stock-id.
  3. **Add a `search_stocks(query)` method** that filters the cached full list by both `stock_id` and `stock_name` (see Issue 3 below).
  4. **Cache TTL for the full list** should be 24h (the entire stock universe changes infrequently).

  ```python
  def _fetch_all_stock_info(self) -> pd.DataFrame:
      """Fetch and cache the entire stock info table (shared across all lookups)."""
      def fetch():
          return self._loader.taiwan_stock_info()
      return self._fetch_or_cache("all_stock_info", fetch)

  def get_stock_info(self, stock_id: str = None) -> pd.DataFrame:
      """Get stock info. Uses shared full-list cache, filters locally."""
      df = self._fetch_all_stock_info()
      if df is None or len(df) == 0:
          return pd.DataFrame()
      if stock_id:
          df = df[df["stock_id"] == stock_id]
      return df

  def search_stocks(self, query: str) -> pd.DataFrame:
      """Search by stock_id or stock_name (exact or partial match)."""
      df = self._fetch_all_stock_info()
      if df is None or len(df) == 0:
          return pd.DataFrame()
      query_lower = query.lower()
      mask = (
          df["stock_id"].str.lower().str.contains(query_lower, na=False)
          | df["stock_name"].str.lower().str.contains(query_lower, na=False)
      )
      return df[mask]
  ```

- **Files Affected**:
  - `src/data/finmind_client.py` — core change
  - `src/pages/_router_base.py` — `get_stock_data()` calls `get_stock_info()` once; no change needed but benefits from the fix
  - Any sidebar search component that calls `get_stock_info()` — should be updated to use `search_stocks()`
- **Impact**: All callers of `get_stock_info()` will automatically benefit. The `search_stocks()` method enables the sidebar name-search feature. No breaking API changes.
- **Priority**: P0

## Issue: Daily Cache Invalidation — `end_date` in Cache Key Causes Guaranteed Miss

- **Layer**: 1
- **Severity**: High
- **Root Cause**: In `get_daily_price()` (line 88–101), `get_monthly_revenue()` (line 103–114), and all other date-range methods, `end_date` defaults to `datetime.now().strftime("%Y-%m-%d")`. This value is passed into `_cache_key()` via `**params`, making the cache key different every day. The `cache_ttl_hours` setting is therefore **completely bypassed** — the cache never hits after midnight. Additionally, old `.json` cache files in `.cache/` are never cleaned up, causing unbounded disk growth.
- **Proposed Fix**:
  1. **Remove `end_date` from the cache key** for methods where the end date is "today". The data for a given `start_date` range doesn't change retroactively (FinMind historical data is append-only). Only include `start_date` and `stock_id` in the key.
  2. **Add a cache cleanup method** that runs on client initialization, removing files older than `cache_ttl`.
  3. **For `get_daily_price` specifically**, the default `end_date` should still be "today" when calling the API (to get the latest data), but it should NOT be part of the cache key. Instead, use a fixed key and rely on TTL for invalidation.

  ```python
  def _cleanup_cache(self):
      """Remove cache files older than cache_ttl."""
      if not self.cache_dir.exists():
          return
      now = datetime.now()
      for f in self.cache_dir.glob("*.json"):
          mtime = datetime.fromtimestamp(f.stat().st_mtime)
          if now - mtime > self.cache_ttl:
              f.unlink(missing_ok=True)

  def __init__(self, cache_dir: str = ".cache", cache_ttl_hours: int = 24):
      self.cache_dir = Path(cache_dir)
      self.cache_dir.mkdir(exist_ok=True)
      self.cache_ttl = timedelta(hours=cache_ttl_hours)
      self._cleanup_cache()  # purge stale files on startup
      # ... rest of init

  def get_daily_price(self, stock_id: str, start_date: str = None, end_date: str = None) -> pd.DataFrame:
      if not start_date:
          start_date = (datetime.now() - timedelta(days=365)).strftime("%Y-%m-%d")
      if not end_date:
          end_date = datetime.now().strftime("%Y-%m-%d")

      def fetch():
          return self._loader.taiwan_stock_daily(
              stock_id=stock_id, start_date=start_date, end_date=end_date
          )

      # Cache key uses only stock_id + start_date (end_date is always "today" and doesn't affect historical data)
      return self._fetch_or_cache("daily_price", fetch,
                                   stock_id=stock_id, start=start_date)
  ```

  Apply the same pattern to `get_monthly_revenue`, `get_per_pbr`, `get_financial_statement`, `get_balance_sheet`, `get_cash_flow`, `get_institutional_investors`, `get_margin_trading`, `get_dividend`, and `get_news` — remove `end_date` from cache keys.

- **Files Affected**:
  - `src/data/finmind_client.py` — all date-range methods
- **Impact**: Cache will actually work across days. Disk usage will be bounded. No API changes. Historical data cached with the old key format will be orphaned (harmless — they'll be cleaned up by the new `_cleanup_cache` on next startup, or can be manually deleted).
- **Priority**: P0

If two events for the same stock have titles sharing the same first 20 characters (e.g., two news articles titled "台積電法說會展望..."), Streamlit raises `DuplicateWidgetID` and crashes the entire page.   title_hash = hashlib.md5(title.encode()).hexdigest()[:8]
  ```python
  if st.button("查看名片", key=f"evt_{stock_id}_{title_hash}"):
  ```

  **Recommended: Option A** — it's simpler, deterministic, and the `enumerate` index is guaranteed unique within the loop iteration.

- **Files Affected**:

- **Impact**: Zero risk. The key is internal to Streamlit and not used for any other purpose. Button functionality is unchanged.

## Issue: Unhandled API Rate Limit — Silent "No Data" Errors

- **Layer**: 2
- **Severity**: Medium
- **Root Cause**: When FinMind's rate limit is hit, the API may return an empty DataFrame, a 429 status, or raise an exception. The current `_fetch_or_cache()` method (line 63–73) treats any empty/None result as "no data" and returns it silently. Users see "Stock ID not found" or "No data" with no indication that the API is rate-limited.
- **Proposed Fix**:
  1. **Detect rate-limit responses** in `_fetch_or_cache` by checking for HTTP 429 or FinMind-specific error messages.
  2. **Raise a custom exception** that callers can catch and display appropriately.
  3. **Add a user-facing warning** in the UI layer.

  ```python
  class FinMindRateLimitError(Exception):
      """Raised when FinMind API rate limit is exceeded."""
      pass

  def _fetch_or_cache(self, prefix: str, fetch_fn, **params) -> pd.DataFrame:
      key = self._cache_key(prefix, **params)
      cached = self._read_cache(key)
      if cached is not None and len(cached) > 0:
          return cached

      try:
          df = fetch_fn()
      except Exception as e:
          error_msg = str(e).lower()
          if "429" in str(e) or "rate" in error_msg or "limit" in error_msg:
              raise FinMindRateLimitError(
                  "FinMind API 速率限制已達上限，請稍後再試。"
              ) from e
          raise

      if df is not None and len(df) > 0:
          self._write_cache(key, df)
      return df
  ```

  Then in UI pages, wrap data loading:

  ```python
  try:
      data = get_stock_data(client, stock_id)
  except FinMindRateLimitError as e:
      st.warning(f"⚠️ {e}")
      st.stop()
  ```

- **Files Affected**:
  - `src/data/finmind_client.py` — add `FinMindRateLimitError` and detection logic
  - All page modules that call FinMindClient methods — add try/except (or handle centrally in `_router_base.py`)
- **Impact**: Users will see a clear message instead of confusing "not found" errors. The custom exception can be caught at the router level to avoid repeating try/except in every page.
- **Priority**: P1

## Issue: Crude ROE Annualization — Quarterly × 4 Is Misleading

- **Layer**: 2
- **Severity**: High
- **Root Cause**: In `src/pages/financial_health.py` line 295, `_calc_roe()` computes `net_income * 4 / equity * 100`. This blindly annualizes a single quarter's net income by multiplying by 4. For companies with high seasonal variance (retail, semiconductors, agriculture), this produces wildly inaccurate ROE values. A company that earns 80% of its profit in Q4 would show an ROE 4× too high if the latest quarter is Q4, or 4× too low if it's Q1.
- **Proposed Fix**: Use **trailing twelve months (TTM)** net income instead of single-quarter × 4:

  ```python
  def _calc_roe(financial_df, balance_sheet_df) -> float:
      """Calculate ROE using TTM (trailing twelve months) net income."""
      if financial_df is None or len(financial_df) == 0:
          return None
      if balance_sheet_df is None or len(balance_sheet_df) == 0:
          return None

      try:
          # Get the 4 most recent quarters for TTM calculation
          net_income = _find_value(financial_df, ["淨利", "本期淨利", "Net Income", "net_income"])

          # Sum the last 4 quarters (TTM) if we have enough data
          sorted_dates = sorted(financial_df["date"].unique(), reverse=True)
          if len(sorted_dates) >= 4:
              ttm_income = 0.0
              for date in sorted_dates[:4]:
                  quarter_data = financial_df[financial_df["date"] == date]
                  val = _find_value(quarter_data, ["淨利", "本期淨利", "Net Income", "net_income"])
                  ttm_income += val
              net_income = ttm_income
          elif len(sorted_dates) >= 1:
              # Fallback: use the single available quarter but DON'T annualize
              # Instead, display as "quarterly ROE" with a note
              latest_date = sorted_dates[0]
              latest_data = financial_df[financial_df["date"] == latest_date]
              net_income = _find_value(latest_date, ["淨利", "本期淨利", "Net Income", "net_income"])

          # Latest equity
          latest_bs_date = balance_sheet_df["date"].max()
          latest_bs = balance_sheet_df[balance_sheet_df["date"] == latest_bs_date]
          equity = _find_value(latest_bs, ["權益總計", "股東權益", "Total Equity", "total_equity"])

          if net_income > 0 and equity > 0:
              return round(net_income / equity * 100, 1)
      except Exception:
          pass

      return None
  ```

  **Key change**: When 4+ quarters are available, sum them for TTM. When fewer than 4 quarters exist, do NOT multiply by 4 — either show the raw quarterly ROE with a "(單季)" label, or return `None` to show "資料不足".

- **Files Affected**:
  - `src/pages/financial_health.py` — `_calc_roe()` function
- **Impact**: ROE values will be accurate for seasonal companies. The display label should indicate "TTM ROE" or "年化 ROE (TTM)" to be transparent. No changes to the analogy engine needed.
- **Priority**: P1

## Issue: Race Conditions in YAML File Operations (Watchlist & Events)

- **Layer**: 3
- **Severity**: Critical
- **Root Cause**: Both `watchlist.py` (`load_watchlist()`/`save_watchlist()`) and `adaptive_engine.py` (`_load_events()`/`_save_events()`) use a read-modify-write pattern with no file locking. Streamlit serves multiple concurrent sessions. If two users simultaneously add to the watchlist, the second write overwrites the first, causing **silent data loss**. Under heavy load, partial writes can also corrupt the YAML file.
- **Proposed Fix**:
  1. **Implement file locking** using `fcntl.flock()` (Unix) or a lock file pattern.
  2. **Create a shared `FileLockManager` utility** in `src/services/file_lock.py` to avoid duplicating logic.
  3. **Use atomic writes** (write to temp file, then rename) to prevent corruption.

  ```python
  # src/services/file_lock.py
  import fcntl
  import os
  import tempfile
  from pathlib import Path
  from contextlib import contextmanager

  @contextmanager
  def atomic_yaml_write(filepath: Path):
      """Context manager for atomic YAML file writes with exclusive locking."""
      filepath = Path(filepath)
      filepath.parent.mkdir(parents=True, exist_ok=True)

      lock_path = filepath.with_suffix(".lock")
      lock_fd = open(lock_path, "w")
      try:
          fcntl.flock(lock_fd, fcntl.LOCK_EX)  # exclusive lock
          yield  # caller reads/modifies data
          # Atomic write: write to temp, then rename
          fd, tmp_path = tempfile.mkstemp(
              dir=filepath.parent, suffix=".tmp"
          )
          try:
              yield tmp_path  # caller writes to tmp_path
              os.replace(tmp_path, filepath)  # atomic on POSIX
          except BaseException:
              os.unlink(tmp_path)
              raise
          finally:
              os.close(fd)
      finally:
          fcntl.flock(lock_fd, fcntl.LOCK_UN)
          lock_fd.close()
          try:
              lock_path.unlink()
          except FileNotFoundError:
              pass
  ```

  Then refactor `save_watchlist()`:

  ```python
  def save_watchlist(entries: list) -> None:
      from src.services.file_lock import atomic_yaml_write
      with atomic_yaml_write(WATCHLIST_PATH) as tmp_path:
          with open(tmp_path, "w", encoding="utf-8") as f:
              yaml.dump(entries, f, allow_unicode=True, default_flow_style=False, sort_keys=False)
  ```

  Apply the same pattern to `_save_events()` in `adaptive_engine.py`.

  **Note**: For Windows compatibility (if needed), replace `fcntl` with `msvcrt` or use the `filelock` pip package (`uv add filelock`).

- **Files Affected**:
  - `src/services/file_lock.py` — new file
  - `src/services/watchlist.py` — `save_watchlist()`, potentially `load_watchlist()` (shared lock)
  - `src/services/adaptive_engine.py` — `_save_events()`, potentially `_load_events()`
- **Impact**: Eliminates data loss and file corruption under concurrent access. The `.lock` files are ephemeral and cleaned up on release. Slight latency increase due to lock acquisition (negligible for this use case).
- **Priority**: P0

## Issue: Fragile Data Field Access in Event Detection

- **Layer**: 3
- **Severity**: Medium
- **Root Cause**: Event detection methods in `adaptive_engine.py` access DataFrame columns directly by hardcoded names: `latest["revenue"]` (line 125), `latest["close"]` (line 200), `prev["close"]` (line 200), `daily_prices.iloc[-1]["date"]` (line 291). If FinMind changes column names (e.g., "close" → "Close", "revenue" → "Revenue"), or if the DataFrame has fewer rows than expected, these raise `KeyError` or `IndexError`. The existing `try/except` blocks catch these but return `None`, silently skipping event detection.
- **Proposed Fix**:
  1. **Create a column name mapping utility** that normalizes FinMind column names to internal names.
  2. **Add explicit validation** before accessing data, with informative logging.
  3. **Use `.get()` with defaults** instead of direct `[]` access where appropriate.

  ```python
  # src/data/column_map.py (new file)
  """Normalize FinMind column names to internal standard names."""

  COLUMN_ALIASES = {
      "close": ["close", "Close", "收盤價", "closing_price"],
      "revenue": ["revenue", "Revenue", "營收", "當月營收"],
      "date": ["date", "Date", "日期", "datetime"],
      "title": ["title", "Title", "新聞標題"],
  }

  def normalize_columns(df) -> pd.DataFrame:
      """Rename FinMind columns to internal standard names."""
      rename_map = {}
      for standard, aliases in COLUMN_ALIASES.items():
          for alias in aliases:
              if alias in df.columns and alias != standard:
                  rename_map[alias] = standard
                  break
      return df.rename(columns=rename_map)

  def safe_get(df, column: str, index: int = -1, default=None):
      """Safely get a value from a DataFrame row by normalized column name."""
      if column not in df.columns:
          return default
      try:
          val = df.iloc[index][column]
          return val if pd.notna(val) else default
      except (IndexError, KeyError):
          return default
  ```

  Then in `adaptive_engine.py`, wrap all FinMind data with `normalize_columns()`:

  ```python
  def detect_revenue_event(monthly_revenue: pd.DataFrame) -> Optional[dict]:
      from src.data.column_map import normalize_columns, safe_get
      monthly_revenue = normalize_columns(monthly_revenue)
      # ... rest of logic using safe_get(monthly_revenue, "revenue", -1)
  ```

- **Files Affected**:
  - `src/data/column_map.py` — new file
  - `src/services/adaptive_engine.py` — all `detect_*` methods and `check_data_freshness`
  - `src/data/finmind_client.py` — optionally normalize at the client level (preferred: normalize once on return)
- **Impact**: If FinMind renames columns, only `COLUMN_ALIASES` needs updating. Event detection becomes resilient to schema changes. The `safe_get` utility prevents `IndexError` on short DataFrames.
- **Priority**: P1

## Issue: Silent Failures in Timeline Filter

- **Layer**: 3
- **Severity**: High
- **Root Cause**: In `src/pages/_router_base.py` line 149–178, `filter_by_timeline()` wraps the entire filtering logic in a `try/except Exception` that returns the **original unfiltered DataFrame** on any error. If the `date_col` has unexpected formatting (e.g., string vs. Timestamp, timezone-aware vs. naive), `pd.to_datetime()` may fail or produce incorrect results. The user sees no error and no filtering — the chart appears to not respond to the 1Y/3Y selector.
- **Proposed Fix**:
  1. **Narrow the exception handling** — catch specific exceptions (`TypeError`, `ValueError`, `KeyError`) rather than bare `Exception`.
  2. **Log the error** for debugging visibility.
  3. **Validate the date column** before attempting conversion.
  4. **Return the original DataFrame only as a last resort**, with a visible warning.

  ```python
  import logging
  logger = logging.getLogger(__name__)

  def filter_by_timeline(df: pd.DataFrame, date_col: str = "date") -> pd.DataFrame:
      if df is None or len(df) == 0:
          return df

      selected = st.session_state.get("timeline_range", "3Y")
      if selected == "ALL":
          return df

      days = _TIMELINE_DAYS.get(selected)
      if days is None:
          return df

      # Validate date column exists
      if date_col not in df.columns:
          logger.warning(f"Timeline filter: column '{date_col}' not found in DataFrame")
          st.warning(f"⚠️ 時間軸過濾失敗：找不到日期欄位 '{date_col}'")
          return df

      try:
          dates = pd.to_datetime(df[date_col], errors="coerce")
          if dates.isna().all():
              logger.warning(f"Timeline filter: all dates in '{date_col}' are NaT after conversion")
              st.warning("⚠️ 時間軸過濾失敗：日期格式無法解析")
              return df

          cutoff = pd.Timestamp.now() - pd.Timedelta(days=days)
          mask = dates >= cutoff
          result = df[mask].reset_index(drop=True)

          if len(result) == 0:
              logger.info(f"Timeline filter: {selected} returned 0 rows (all data before cutoff {cutoff.date()})")
              st.info(f"ℹ️ 選擇的時間範圍 ({selected}) 內無資料")

          return result

      except (TypeError, ValueError, KeyError) as e:
          logger.error(f"Timeline filter error: {e}")
          st.warning(f"⚠️ 時間軸過濾失敗：{str(e)}")
          return df
  ```

- **Files Affected**:
  - `src/pages/_router_base.py` — `filter_by_timeline()` function
- **Impact**: Users will see a warning instead of silent failure. The `errors="coerce"` parameter in `pd.to_datetime` prevents crashes on malformed dates. The `st.info` for empty results clarifies that the filter worked but there's no data in range.
- **Priority**: P1
