### Justification

Sprint 18 delivered 6 design-related changes that collectively represent the most design-cohesive sprint since C105 (Sprint 10). Every feature was built using shared components from `_router_base.py`, zero inline HTML regressions were introduced, and the new `_explain_button()` and `_so_what_box()` helpers establish reusable patterns for future features.

**Positive findings:**

1. **C139 Explain This Number 💡 buttons — Excellent**
   - The `_explain_button()` helper in `_router_base.py` (lines 88-130) is a **well-designed shared component**. It accepts `metric_name`, `metric_value`, `delta`, `context`, `key_prefix`, and `source_label` parameters — a clean API that covers all use cases.
   - Uses `st.popover("💡")` with `help` tooltip for accessibility. The popover pattern is proven (from `_render_metric_popover()` in `_financial.py`) and works well for the "explain this" UX.
   - Graceful degradation: the `except Exception` fallback shows the metric name/value with "暫時無法產生解釋" — no crash, no broken UI.
   - Deployed consistently across 4 section files: `_summary.py` (story card metrics), `_health.py` (5 health dimensions), `_moat.py` (5 moat dimensions), `_story.py` (delta metrics). This is exactly the cross-page reuse that D-003 demands.
   - The `key_prefix` parameter ensures unique Streamlit keys following the design system convention (`{function}_{stock_id}`).
   - **Ten-second test**: A user sees a 💡 button next to any metric, clicks it, and gets a plain-language explanation in <10 seconds. The popover content (display name → value → explanation → source badge) follows a clear visual hierarchy.

2. **C141 Source Badge — Minimalist and correct**
   - Uses `st.caption(source_label)` at the bottom of the `_explain_button()` popover. This is the right design decision — visible but not competing with content.
   - Two badge variants: `"📊 FinMind"` for API-sourced data, `"📊 系統估算"` for template-generated explanations. This distinction builds trust transparency from day one.
   - No new component needed — the source badge is baked into `_explain_button()` as a parameter. This is good API design.
   - **Competitive alignment**: Public.com uses "Based on company filings" in 0.7rem text. Finimize uses a confidence bar. Our `st.caption()` approach is the right level of subtlety for the historian positioning.

3. **C143 Implication Sentence — Well-executed with proper tone controls**
   - The `DeltaExplanationProvider.explain()` method now returns an `implication` field in `ExplanationResponse`. The implication is generated by `_pick_implication()` which uses tiered templates keyed on `metric_name × direction`.
   - **Tone compliance**: The implication templates in `_IMPLICATION_TEMPLATES` (lines 79-116 of `delta_explanation_provider.py`) use safe alternatives: "可留意", "可觀察", "可持續追蹤" instead of blocklisted terms like "值得關注", "需要密切關注".
   - The delta card rendering in `_story.py` (lines 31-36) replaces the old explanation text with the implication sentence on the card face, while the full explanation moves to the 💡 popover. This is the right information hierarchy — the implication is the takeaway, the explanation is one click away.
   - **Design system compliance**: The implication sentence is rendered inside `_info_card()` as part of the delta text block. No new card type needed. The colored span (`#27AE60`/`#E74C3C`) for the delta value is consistent with the design system's positive/negative colors.

4. **C149 So What? Box — Clean synthesis component**
   - The `_so_what_box()` helper in `_router_base.py` (lines 184-216) is a well-designed shared component. It synthesizes multiple delta implications into a single callout.
   - **Visual design**: Uses `background:#F0F7FF` (light blue) and `border-left:4px solid #2980B9` (medium blue) — a distinct visual treatment that differentiates it from `_info_card()` (gray/blue) and `_summary_card()` (orange/amber). This is a new "synthesis card" variant that should be documented in the design system.
   - **Conditional rendering**: Only shown when there are 2+ active deltas (`if len(deltas) < 2: return`). This prevents the box from appearing with a single implication, which would be redundant.
   - **Content synthesis**: Joins multiple implications with "；" (Chinese semicolon) for natural reading. The "🧭 所以呢？" header is conversational and beginner-friendly.
   - **Competitive alignment**: Stockstory's "💡 So What?" box uses a similar pattern (distinct visual card after metric sections). Our implementation is more conservative (blue vs. amber) which fits the historian positioning.

5. **D-097 Tone QA Audit — Systematic and thorough**
   - The `tests/test_tone_qa.py` file (325 lines) implements an AST-based blocklist scanner that checks all string literals in `src/services/` and `src/pages/` against the tone blocklist.
   - **Blocklist coverage**: 11 terms (建議, 應該, 買, 賣, 推薦, 進場, 出場, 值得關注, 需要密切關注, 值得持續追蹤, 表現優於預期) — expanded from the original Discussion Round 38 set.
   - **Exclusion handling**: The `_EXCLUDED_FILES` dict (30+ entries) properly categorizes files with pre-existing violations (EDU, OOS, TPL, CHART, DISCLAIMER). Each entry has a reason code and follow-up task reference. This is a mature approach — rather than ignoring violations, they're tracked and scheduled.
   - **Allowed contexts**: The `ALLOWED_CONTEXTS` list handles safe uses (e.g., "大賣" = "sold well" in a revenue template, "買入" in a PE ratio template). This prevents false positives.
   - **CI integration**: Marked with `@pytest.mark.tone` for independent running. The test outputs scan stats on pass and detailed violation info on fail.

6. **Overall design discipline — Zero inline HTML regressions**
   - All Sprint 18 features use shared components from `_router_base.py`. No new `unsafe_allow_html=True` instances were introduced in the feature code.
   - The `_explain_button()` and `_so_what_box()` helpers are properly placed in `_router_base.py` (the shared component library) rather than being duplicated in each section file.

**Remaining concerns (not grade-blocking):**

- **D-108 (still open)**: Settings page visual feedback boxes use inline HTML. This is a pre-existing issue from Sprint 17, not a Sprint 18 regression. The fix (create `_feedback_box()` or use `_info_card()`) is still pending.
- **D-109 (still open)**: Benchmark logic duplicated in `_summary.py`. Also pre-existing from Sprint 17. The fix (extract to shared service) is still pending.
- **D-110 (NEW)**: `_so_what_box()` uses a new color (`#F0F7FF` background, `#2980B9` border) not in the design system. See new debt section below.
- **D-111 (NEW)**: `_explain_button()` popover uses `st.caption()` for source badge, which renders at Streamlit's default caption size (~0.8rem). This is slightly smaller than the design system's label spec (0.85rem). Minor inconsistency.

**Net assessment**: Sprint 18 is the most design-disciplined sprint in recent rounds. The `_explain_button()` and `_so_what_box()` components are well-designed, properly placed in `_router_base.py`, and reused across 4+ section files. The tone QA automation is a significant quality gate addition. The grade is maintained at **A** for the 6th consecutive review.

## 3. New Design Debt Items (D-112+)

### D-112: `_so_what_box()` Uses Color Not in Design System

- **Severity**: P2
- **Added**: 2026-06-14
- **Source**: Design Review Round 39
- **Description**: The `_so_what_box()` helper in `_router_base.py` (line 212) uses `background:#F0F7FF` and `border-left:4px solid #2980B9`. The design system defines only 3 card background colors: `#F8F9FA` (info), `#FFF8F0` (tip/summary), and `white` (subsidiary/entity). The `#F0F7FF` light blue background is a new color variant not documented in the design system. Similarly, `#2980B9` (medium blue) is a new border color — the design system uses `#3498DB` (primary blue), `#F39C12` (amber), `#27AE60` (green), `#E74C3C` (red).
- **Affected Files**: `src/pages/_router_base.py` line 212
- **Proposed Fix**: Either (a) change to an existing design system color (e.g., `background:#F8F9FA` with `border-left:4px solid #3498DB` to match `_info_card()`), or (b) document the new "synthesis card" variant in the design system with its own color spec. Option (b) is recommended since the "So What?" box is a distinct visual element that should look different from info cards.
- **Effort**: 0.5h (documentation update + color decision)

### D-113: `_explain_button()` Source Badge Uses `st.caption()` at 0.8rem vs Design System 0.85rem

- **Severity**: P2
- **Added**: 2026-06-14
- **Source**: Design Review Round 39
- **Description**: The source badge in `_explain_button()` (line 124 of `_router_base.py`) uses `st.caption(source_label)` which renders at Streamlit's default caption font size (~0.8rem). The design system specifies labels at `font-size: 0.85rem`. This is a minor inconsistency — the source badge is 0.05rem smaller than the design system spec.
- **Affected Files**: `src/pages/_router_base.py` lines 124, 130
- **Proposed Fix**: Replace `st.caption()` with `st.markdown(f'<span style="font-size:0.85rem;color:#7F8C8D;">{source_label}</span>', unsafe_allow_html=True)` to match the design system's label spec exactly.
- **Effort**: <0.25h (two-line change)

### D-114: `_explain_button()` Popover Content Doesn't Follow Card Visual Hierarchy

- **Severity**: P2
- **Added**: 2026-06-14
- **Source**: Design Review Round 39
- **Description**: The popover content in `_explain_button()` renders: (1) `**display_name**` (bold), (2) `_value_text_` (italic), (3) `---` (divider), (4) `explanation_text` (plain). This doesn't follow the design system's card visual hierarchy which specifies: label (0.85rem, gray) → value (1.6rem, bold) → plain-language (0.85rem, green, italic). The popover uses markdown bold/italic instead of the card component's structured layout.
- **Affected Files**: `src/pages/_router_base.py` lines 120-123
- **Proposed Fix**: This is a trade-off — popovers have limited space and can't render full card layouts. The current approach is acceptable for a popover context. Document as a known limitation: "Popover explanations use markdown formatting instead of card hierarchy due to space constraints."
- **Effort**: 0.25h (documentation only — no code change recommended)

### D-115: Tone QA Test Excludes 30+ Files with Pre-existing Violations

- **Severity**: P2
- **Added**: 2026-06-14
- **Source**: Design Review Round 39
- **Description**: The `test_tone_qa.py` excludes 30+ files via `_EXCLUDED_FILES` (lines 83-162). While each exclusion has a reason code and follow-up task reference, this means the tone blocklist is not enforced on a significant portion of the codebase. The excluded files include `analogy_engine.py`, `metric_education.py`, `news_summarizer.py`, `story_feed.py`, `event_interpretation_service.py`, `risk_analyzer.py`, and 20+ page files. If these files contain user-facing strings with blocklist terms, they won't be caught by CI.
- **Affected Files**: `tests/test_tone_qa.py` lines 83-162
- **Proposed Fix**: Create a prioritized audit plan for the excluded files. Start with `src/services/` files (higher risk — they generate user-facing content) before `src/pages/` files. Track as Sprint 19 debt clearance items.
- **Effort**: 4-6h (full audit of all excluded files)

### D-116: `_so_what_box()` Minimum Delta Threshold Not Validated in Component

- **Severity**: P2
- **Added**: 2026-06-14
- **Source**: Design Review Round 39
- **Description**: `_so_what_box()` guards against single-delta rendering (`if len(deltas) < 2: return`) but doesn't validate that the delta dicts contain the required keys (`implication`, `direction`, etc.). If a caller passes a malformed delta dict, the component will render an empty box (all implications are empty strings) without any error or fallback.
- **Affected Files**: `src/pages/_router_base.py` lines 184-216
- **Proposed Fix**: Add validation: check that at least one delta has a non-empty `implication` before rendering. If all implications are empty, return silently (don't render the box).
- **Effort**: 0.25h

## 5. Design Recommendations for Sprint 19

### Recommendation 1: Document `_so_what_box()` as "Synthesis Card" Variant in Design System (Priority: 🟢 Quick win, 0.5h)

The `_so_what_box()` uses a new color scheme (`#F0F7FF` background, `#2980B9` border) that should be documented in the design system as a "synthesis card" variant. This is a distinct visual element that summarizes multiple data points into a single callout — different from info cards (single fact) and summary cards (key takeaway).

**Action**: Add to `docs/design/design_system.md` Section 3.3:
```html
<!-- Synthesis card (light blue, implies connection between data points) -->
<div style="background:#F0F7FA;border-radius:12px;padding:1.2rem;border-left:4px solid #2980B9;margin:0.8rem 0 0.5rem 0;">
    <div style="font-weight:600;color:#2C3E50;">{icon} {title}</div>
    <div style="font-size:0.9rem;color:#2C3E50;margin-top:0.4rem;line-height:1.7;">{synthesized_content}</div>
</div>
```

### Recommendation 2: Fix D-108 and D-109 as Sprint 19 Day 1 Tasks (Priority: 🟢 Quick wins, 1.5-3h combined)

Both D-108 (settings page inline HTML) and D-109 (benchmark logic duplication) are P2 items identified in Round 38 that are still open. They're well-scoped, low-risk fixes that should be done before new feature work begins.

- **D-108**: Replace `settings.py` inline HTML feedback boxes with `_info_card()` or a new `_feedback_box()` helper. Effort: 0.5-1h.
- **D-109**: Extract benchmark health score fetching from `_summary.py` (lines 183-288) into a shared service function that both `_summary.py` and `_health.py` can call. Effort: 1-2h.

### Recommendation 3: Expand Tone QA Coverage to Excluded `src/services/` Files (Priority: 🟡 Important, 2-3h)

The 30+ excluded files in `test_tone_qa.py` include several `src/services/` files that generate user-facing content: `analogy_engine.py`, `metric_education.py`, `news_summarizer.py`, `story_feed.py`, `event_interpretation_service.py`, `risk_analyzer.py`. These should be audited and brought under tone QA coverage in priority order.

**Action**: 
1. Audit `analogy_engine.py` and `metric_education.py` for blocklist violations (1h)
2. Rewrite any violating strings to use safe alternatives (0.5-1h)
3. Remove the files from `_EXCLUDED_FILES` and verify tests pass (0.5h)
4. Create follow-up tasks for remaining excluded files

### Recommendation 4: Wire `_glossary_tooltip()` to `_explain_button()` Popover (Priority: 🟡 Important, 1-2h)

The `_glossary_tooltip()` component in `_router_base.py` (lines 269-290) is fully built but not wired to any page (D-091, still open). The `_explain_button()` popover is the natural integration point — when the explanation text contains financial terms (PER, ROE, 毛利率, etc.), those terms should be tappable for definitions.

**Action**:
1. Add an optional `glossary_terms` parameter to `_explain_button()` — a list of `(term_key, display_name)` tuples
2. In the popover, after rendering the explanation text, render a "📖 相關術語" section with `_glossary_tooltip()` for each term
3. Wire to the story card metrics in `_summary.py` (PER, ROE, 毛利率 are all glossary terms)

### Recommendation 5: Standardize Popover Content Layout Across All Explanation Types (Priority: 📋 Sprint 19+, 1h)

Currently, there are three different popover content layouts in the codebase:
1. `_explain_button()` — bold name → italic value → divider → explanation → caption
2. `_glossary_tooltip()` → bold name → italic definition → example → analogy
3. `_render_metric_popover()` in `_financial.py` — uses `_白话_card()` inside a column

These should be standardized to a single popover content pattern for consistency.

**Action**: Create a `_popover_content(title, subtitle, body, footer)` helper in `_router_base.py` that renders a consistent layout. Refactor `_explain_button()`, `_glossary_tooltip()`, and `_render_metric_popover()` to use this helper.
